# Topic 12 — Capability Routing, Fallback Models, Escalation, and Heterogeneous Model Portfolios

## 1. Problem and objective

Topic 11 built the static selection table; this topic makes it a running system. Four mechanisms constitute portfolio operation: **routing** (per-task model assignment, online), **fallback** (automatic substitution on failure), **escalation** (deliberate upgrade to a stronger or differently-authorized resource, including humans), and the **portfolio** itself (the managed set these mechanisms operate over). The objective is the full machinery as the strongest source specifies it — the C-A-F loop, its bandit formalization, and the three-module router that won on cumulative regret — plus the fallback and escalation semantics the runtime and deployment sources document, and the failure analysis for the layer that now sits between your application and every model it uses.

## 2. Intuition first

A routing layer is a dispatcher with a memory. The naive dispatcher reads the ticket and guesses the right specialist — and the ablation already told us that guessing from descriptions performs *below* not guessing at all (41.18 vs 41.41 [AAR Table 1]). The competent dispatcher tracks what actually happened on every past assignment, verified by results rather than by the specialists' self-reports, and gets measurably close to omniscient assignment. The whole topic is that sentence, formalized — plus the two safety valves every dispatcher needs: what to do when the assigned specialist fails mid-job (fallback), and when to recognize the job exceeds the roster entirely (escalation).

## 3. The C-A-F loop, formally

The source's formalization deserves exact statement [AAR §3.2]. The router faces a model pool 𝓜 = {m_1, …, m_M} and a task stream 𝓣 = (t_1, …, t_N). At task t_i:

```
c_i ──Decide──▶ a_i ──Execute──▶ f_i ──Memorize──▶ c_{i+1}

Context  c_i = (p_i, d_i, 𝓗_{<i})   — prompt, optional metadata, accumulated memory
Action   a_i ∈ [M]                   — which model to invoke
Feedback f_i = (ŝ_i, κ̂_i)           — verifier-observed score ∈ [0,1] and monetary cost
```

Per-task reward combines both under user-set weights: r_i(a_i) = ε₁·s_i(a_i) + ε₂·κ_i(a_i), with ε₁ > 0, ε₂ < 0 [AAR eq. 2]. The problem is a contextual multi-armed bandit; the benchmark metric is **cumulative regret** against the per-task oracle: CumReg_N(π) = N(V* − V(π)), where V* averages the per-task maximum reward [AAR eqs. 5–7]. Two design theorems fall out of the diagnosis, stated by the authors as requirements: the router must "*acquire* new execution-grounded information at each decision — performance signals generated by actually running the selected model's output in a sandbox rather than relying on static priors or model self-assessment — and *accumulate* it across the task stream so that future decisions can condition on past outcomes" [AAR §3.1]. Acquisition and accumulation: a Verifier and a Memory. Static routers, lacking both, "are structurally unable" to close the information gap [AAR §1].

## 4. The reference architecture: ACRouter

The instantiation, module by module [AAR §3.3]:

- **Orchestrator (integrating):** routes on dynamic context — "the DimensionBest prior, the top-10 historical neighbors retrieved from Memory by kNN, and task metadata"; the core policy is "a cost-effective Qwen3.5-0.8B model fine-tuned on the CodeRouterBench probing set, combined with heuristic rules via weighted voting." Note every design choice: priors from measurement, neighbors from experience, and a *small fine-tuned* decision model — the routing decision itself does not need frontier intelligence (Topic 11 §3.2), it needs the right inputs.
- **Verifier (producing):** scores each outcome by aggregating typed verification tools — "AST parsing and sandbox execution" — as u_i = Σ_k w_{d(t_i),k}·ŝ_k with type-specific weights summing to 1 [AAR eq. 8]. Execution-grounded, multi-signal, self-assessment excluded.
- **Memory (accumulating):** the store the Orchestrator's kNN retrieval runs against — experience as an index, not a vibe.

Results, with the honest scoping: ACRouter "attains the lowest cumulative regret across all evaluated routers on in-distribution streams and also generalizes well to OOD tasks, consistently outperforming other routers," while "lightweight static routers fail to generalize on OOD tasks" [AAR §1, contributions]. The architecture is domain-general; the evidence is coding-domain (Topic 11 §8's caveat carries over).

## 5. Fallback and escalation: the safety valves

**Fallback** is substitution under failure, and the sources document it at three distinct layers, worth separating because their semantics differ:

1. *Within-request fallback:* the runtime's structured-output path includes "a model fallback [that] retracted the completed output with no successful retry" as a typed terminal condition [CAL] — substitution can *undo* work, and callers must treat pre-fallback outputs as uncommitted (Topic 7 §5.2).
2. *Policy-level fallback:* the Fable 5 deployment is a production fallback architecture — "Fable 5's cybersecurity classifiers are effective at detecting cyber use and cause the model to fall back to Opus 4.8," so "Fable 5 performs similarly to that model" in the triggered domains [FSC Exec. Summary]. Read the design: a *classifier* (not the model's judgment) triggers substitution to a *known-safe* (not merely available) alternative, and the vendor reports the composite's behavior per-regime. That is the pattern to copy.
3. *Availability fallback:* provider-outage substitution — operationally necessary, unmeasured in this ledger, and dangerous by default because the substitute's propensity profile (Topic 11 §4-Risk) was not the one qualified. Rule: availability fallback only within the same qualified risk class, else fail toward escalation.

**Escalation** differs from fallback in direction and trigger: it is a *deliberate* upgrade — to a stronger model, higher effort (the per-subagent effort override is the in-family escalation mechanism [CAL]), or a human — triggered by uncertainty and consequence rather than by failure. Topic 8 §4's routing gate is the decision point: `verify-more → escalate → abstain` is an ordered ladder, and the human rung is a *feature of the architecture*, not an SLA breach. Escalation to humans inherits one documented hazard worth repeating here: the approval channel is a surface the model can learn to optimize against [FSC §2.3.3.3]; escalation requests are model outputs, and reviewers should see evidence (traces, diffs), not just the model's summary of why approval is warranted.

## 6. Portfolio operations

The portfolio is the managed asset the mechanisms run over; its lifecycle is the selection table (Topic 11 §6) operated continuously **[derived — synthesis; components sourced]**:

- **Admission:** a candidate model enters after probing-set measurement across the five dimensions, including propensity bounds [Topic 11 §5's constraints].
- **Assignment drift monitoring:** the router's regret and per-class routing distribution are standing metrics; a model whose routing share collapses is dead weight (remove it — namespace hygiene applies to model pools as to tools), and a sudden share shift flags either drift in the stream or a version change in a member.
- **Version churn:** every member update re-triggers the propensity and performance checks — both cards' version-over-version regressions [G56 §1; FSC §6.3.5.2] arrived within stable product names, and the router's Memory now contains *stale experience* about the renamed model; experience stores need version keys, or accumulated evidence becomes accumulated error.
- **Tiering:** the provider family structure (flagship / lower-cost / fastest [G56 §1]) maps onto escalation ladders naturally: route to the cheapest tier whose class-conditional numbers clear the bar; escalate up-tier on the uncertainty gate.

## 7. Failure modes

- **Static router rot:** the measured OOD failure of lightweight static routers [AAR §1]; a frozen policy meets a drifting stream.
- **Self-assessment leakage into feedback:** the loop's integrity rests on verifier-observed scores [AAR §3.1]; letting the routed model grade its own output re-imports every Topic 8 §3.1 distortion into the router's training signal.
- **Memory poisoning across versions** (§6.3): un-versioned experience misattributing an old model's record to its changed successor.
- **Fallback class violations** (§5.3): availability substitution across risk classes; the incident report writes itself.
- **Retraction blindness:** consuming within-request outputs that a fallback later retracted [CAL]; commit only on terminal success.
- **Escalation starvation:** thresholds tuned so humans are never bothered — indistinguishable, from the metrics, from a system with no escalation at all; audit the escalation *rate* against the task stream's known-hard fraction.
- **Router as single point of failure:** the dispatcher now fronts everything; its latency adds to every task and its outage is total. The router needs the same budget/timeout/bypass engineering as any gateway (Chapter 14), including a qualified default-model bypass path.

## 8. Limitations

- The quantitative case is one domain (coding), one benchmark family (CodeRouterBench, ~10K tasks, 8 models [AAR]); regret magnitudes elsewhere are unmeasured. The *structure* — verified feedback beats priors beats descriptions — is the transportable finding.
- ε₁/ε₂ are user-set weights [AAR eq. 2]; the framework prices trade-offs but does not choose them — the business owns ε₂, and pretending otherwise hides a decision inside a hyperparameter.
- Verifier construction is the hidden cost: execution-grounded scoring presumes sandboxes and oracles (cheap in coding, expensive elsewhere — Chapter 1 Topic 11's oracle-quality regularity); where verification is judge-based, the router's feedback inherits judge bias (Chapter 13).
- Human-escalation design is under-evidenced in this ledger beyond the hazard note [FSC §2.3.3.3]; Chapter 12's approval-policy machinery carries it.

## 9. Production implications

1. **Build the loop before the router is clever:** Verifier and Memory first — the ablation ordering (statistics > descriptions > nothing) says the *inputs* are the capability [AAR Table 1]; a lookup table over verified history beats a frontier model routing on vibes.
2. **Keep the routing model small and the evidence rich** — the reference design routes with a 0.8B fine-tune over measured priors and kNN experience [AAR §3.3]; spend frontier tokens on tasks, not on dispatch.
3. **Copy the classifier-triggered fallback pattern** [FSC]: deterministic trigger, pre-qualified target, per-regime reporting — for every fallback path, know and publish which regime the composite is in.
4. **Wire the escalation ladder to the uncertainty gate** (Topic 8 §4) with consequence-class thresholds, and audit both directions: escalation rate (starvation) and approval-evidence quality (the [FSC §2.3.3.3] channel).
5. **Version-key everything the loop accumulates** (§6.3) and re-qualify members on update — the portfolio's memory is only as good as its provenance.
6. **Report portfolio health as regret**, not as per-model win rates: CumReg against your own probing-set oracle [AAR eq. 7] is the single number that says whether the dispatch layer is earning its latency.
