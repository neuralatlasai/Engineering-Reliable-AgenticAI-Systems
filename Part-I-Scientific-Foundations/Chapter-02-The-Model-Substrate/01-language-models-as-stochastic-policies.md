# Topic 1 — Language Models as Stochastic Policies Rather Than Transaction Processors

## 1. Problem and objective

Most production software is built from components with transactional semantics: deterministic given inputs, idempotent under retry, atomic in effect, and honest about failure (an exception, not a plausible-looking wrong answer). Engineers' intuitions — and their retry loops, their caching layers, their test suites — assume those semantics. A language model provides none of them. The objective of this topic is to state exactly which guarantees π_M does and does not offer, ground each absence in the sources, and derive the design consequences before the rest of the chapter builds on them.

## 2. Intuition first

Call a payments API twice with the same arguments and you have either one charge (idempotency) or a detectable error. Call a model twice with the same context and you have two *samples* from a distribution — possibly agreeing, possibly not, neither one flagged as the outlier. The model is not a function; it is a policy you query. That is not a defect to be patched — sampling is where adaptive behavior comes from — but it means every transactional property your system needs must be *manufactured around* the model, never assumed of it. Chapter 1 called the manufacturing layer the harness; this topic is the precise inventory of what it must manufacture.

## 3. Formalization: what the component actually is

From the formal model [MEM §2.1]: the agent's action is a_t = π_i(o_t^i, m_t^i, 𝒬), where all five action types — language, tool invocation, planning, environment control, communication — "are produced through an autoregressive LLM backbone conditioned on a contextual input," and the policy "may internally generate multi-step reasoning chains, latent deliberation, or scratchpad computations prior to emitting an executable action; such internal processes are abstracted away."

A policy, formally, induces a distribution over actions given context — the router literature writes this explicitly as π(· | h) ∈ Δ over the choice set [AAR §3.2]. Three properties follow:

1. **Nondeterminism:** repeated queries are i.i.d.-ish draws, not repeated evaluations. Retry changes the answer; that is simultaneously the recovery mechanism of Topic 1.8's verification algebra (fresh draw after detected failure) and the reproducibility hazard of Chapter 13 (a benchmark number is a sample statistic).
2. **No failure channel in-band:** a wrong sample is syntactically indistinguishable from a right one. Contrast a transaction processor, whose failures are typed. The model's "error signal" must be constructed outside it — which is the entire justification for external verification (Topic 1.2, invariant I4).
3. **Statelessness:** "a harness turns a *stateless* language model into a functional agent by grounding its outputs in external execution, persistent state, and verifiable feedback" [CAH §2]. Whatever continuity exists lives in context and storage, not in the component.

## 4. The guarantee inventory

| Transactional property | Model API provides? | What manufactures it, if needed |
|---|---|---|
| Determinism | No — sampled outputs | Pin nothing on it; design for distribution (voting, verification) |
| Idempotency | No — re-query = new draw | Idempotency keys and dedup at the *tool/action* layer (Ch. 5) |
| Atomicity | No — partial/interleaved outputs possible | Turn-level transaction boundaries in the harness [CAL] |
| Typed failure | No — wrong answers are well-formed | External validators, oracles, judges [HB §3.4; AAR §3.3] |
| Faithful computation | No — "unreliable at faithfully carrying out symbolic, logical, or arithmetic computation" | Delegate computation to executed code [CAH §2.1] |
| Honest self-report | No — self-assessment is policy output | Never grade π by π; sandbox-execute instead of trusting self-assessment [AAR §3.1; FSC §2.3.3] |

The last two rows deserve their evidence stated plainly. On computation: the Code-as-Agent-Harness survey's founding observation is that models "remain unreliable at faithfully carrying out symbolic, logical, or arithmetic computation" even when effective at *proposing* steps — hence program-delegated reasoning, where "the model proposes procedures, while the harness executes them" [CAH §2.1]. On self-report: ACRouter's designers explicitly rejected model self-assessment as a feedback source, requiring "performance signals generated by actually running the selected model's output in a sandbox rather than relying on static priors or model self-assessment" [AAR §3.1]; and the system card documents why — a frontier model that "says it tested work end to end, when it had not" [FSC §2.3.3.2].

## 5. The stochasticity is load-bearing, not incidental

It would be a mistake to read this topic as "models are broken transaction processors." The distribution *is* the capability:

- **Retry-with-fresh-draw** is what makes detection-plus-retry (Topic 1.8 §4) improve success at all; a deterministic component would fail identically forever.
- **Sampling diversity** powers the parallelization-with-voting workflow pattern [BEA] and search-based planning, which "allocates inference-time compute to systematically explore, evaluate, and select among multiple candidate solution paths... increasing conceptual diversity before implementation" [CAH §3.1.3].
- **Distributional breadth** is what absorbs input variation at workflow leaves (Chapter 1, Topic 9's W2 condition) without enumerated branches.

The engineering stance is therefore not to suppress stochasticity but to *place* it: inside verified loops and diversity-consuming aggregators, never on unverified irreversible paths. This is Chapter 1's three-layer stack (Topic 1.4) restated from the component's point of view.

## 6. Consequences for system design

1. **Every model output is a proposal.** The type of π_M's output channel is "candidate action," and the harness's admission decision (permissions, hooks, validation [CAL]) is what converts a proposal into an action. Systems that wire model output directly to effectful interfaces have silently assigned transaction semantics to a sampler.
2. **Retries are semantic events, not transport events.** An HTTP retry replays a request; a model retry re-decides. Retry policies must therefore be written at the level of *task state* (what has been verified done) rather than request state — Chapter 10's restart-safe execution in embryo.
3. **Caching changes behavior.** Caching a deterministic function is transparent; caching a policy's draw pins one sample forever. Legitimate (prompt caching of *inputs* [CAL]) and illegitimate (caching outputs as if canonical) caching must be distinguished (Chapter 14).
4. **Reproducibility is statistical.** Any claim about the model's behavior is a claim about a distribution: report sample sizes and variance (Chapter 1, Topic 12 §4), never single-run anecdotes.
5. **The equivalence class of "same input" is fragile.** The policy conditions on the *entire* assembled context; CompWoB's instruction-order sensitivity — performance "further deteriorates when task instruction order changes" [CompWoB] — shows that inputs an engineer considers equivalent, the policy does not.

## 7. Failure modes from ignoring the distinction

- **The silent-wrong-answer pipeline:** downstream code consumes model output as if typed-or-throw; the wrong sample propagates as state (Topic 1.8's contamination). Documented at frontier: uncritically executing a user's subtly incorrect command example, then correcting *after* execution, where the prior model checked documentation first [FSC §6.3.5.4].
- **Retry storms that re-sample the same failure:** when the error cause lives in the context (bad evidence, poisoned instruction), fresh draws share it; retries burn budget without independence. Detection must distinguish *sampling* failures (retry helps) from *conditioning* failures (retry cannot).
- **Test flakiness misdiagnosed:** treating run-to-run variance as infrastructure noise and "fixing" it with tighter seeds hides the distribution the production system will actually face.
- **Anthropomorphic debugging:** asking the model why it did something and treating the answer as telemetry. The stopping-decision evidence — visible text never mentioning the internal "budget exhausted"/"I'm tired" states that NLA decodings surfaced [FSC §6.4.1.4] — is the definitive counterexample: verbalized rationale and operative cause are separate variables.

## 8. Limitations

- The clean policy abstraction hides temperature, sampling configuration, and provider-side changes; "the distribution" is itself versioned and drifts with model updates — Chapter 1's re-qualification rule exists for this reason [HB §4.3].
- The sources do not quantify output variance for current frontier models on agentic tasks (Harness-Bench's variance analysis is cross-harness, not cross-seed [HB §4.3]); per-seed variance on your task suite is a measurement you must make, not look up.
- The claim "self-report is untrustworthy" is directionally certain but not uniform: the diligence evaluations show current models proactively surfacing flaws at high rates in *some* regimes (misreported-rate 0.000 for Mythos 5 on planted-flaw analysis [FSC §6.3.5.1]) — the failure is conditional and environment-dependent, which is worse for prediction, not better.

## 9. Production implications

1. Write the guarantee inventory (§4) into interface contracts: every consumer of model output must name which absent property it compensates for, and how.
2. Budget verification as a first-class cost center — it is the price of using a sampler as a component, and Topic 1.8 showed it dominates model upgrades on long tasks.
3. Instrument *distributional* health: agreement rates across samples, retry-success rates split by suspected cause (sampling vs. conditioning), variance trends across model versions.
4. Prohibit, by architecture review rule, any direct model-output-to-irreversible-action path (Chapter 1, Topic 6's reversibility gates).
